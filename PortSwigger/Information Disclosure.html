Information Disclosure

"Every extra detail revealed is a foothold for the attacker."

Overview

Information disclosure occurs when an application unintentionally exposes sensitive details to users or attackers. These leaks may seem harmless at first (debug data, error messages, metadata), but they often reveal crucial information such as file paths, stack traces, source code, credentials, or even keys for authentication.

The following PortSwigger labs demonstrate how small leaks can escalate into major vulnerabilities, sometimes even enabling authentication bypass or full compromise.

Labs

  • Information disclosure in error messages : Error handling reveals details like stack traces, file paths, database queries, or internal server info.
  • Information disclosure on debug page : Exposed debugging endpoints provide sensitive configuration values, environment variables, or credentials.
  • Source code disclosure via backup files : Forgotten or misconfigured backup files (.bak, ~, old versions) expose the underlying source code and secrets.
  • Authentication bypass via information disclosure : Leaked details allow prediction or manipulation of authentication tokens, enabling bypass of login controls.
  • Information disclosure in version control history : Publicly accessible repositories or exposed .git directories reveal commit history, including sensitive files and hardcoded credentials.

Key Takeaways

  • Seemingly harmless information can be weaponized: every leak lowers the attacker’s workload.
  • Common culprits include error messages, debug endpoints, backup files, and version control metadata.
  • Developers should enforce strict error handling, never expose debug functionality in production, and clean up artifacts such as backups or .git folders.
  • Information disclosure often chains into other attacks, such as authentication bypass, privilege escalation, or RCE.

Information disclosure is not noise, it’s signal. Treat every leak as an attack surface.
