Bug Bounty Report

"I prefer stealth over spectacle."

Introduction

Dans le contexte du bug bounty, le rapport est la clé de votre reconnaissance et de votre récompense. Ce n'est pas un document formel comme dans un pentest, mais un compte-rendu concis, clair et reproductible à 100% pour les équipes de tri et de sécurité du programme.

Spécificités d’un rapport de bug bounty

  • Il s’adresse à des triagers techniques, souvent surchargés
  • Il doit être clair, vérifiable rapidement et bien structuré
  • La priorité (ou sévérité) peut être discutée après soumission
  • Un bon rapport augmente les chances de duplication évitée (et donc d’avoir la prime)

Structure recommandée

1. Titre explicite

Exemples : Stored XSS on user profile page, IDOR on /api/user/export endpoint.

2. Impact immédiat

Dans une ou deux phrases : quel est le risque ? Est-ce que des données sensibles sont accessibles ? Peut-on escalader ? Gagner des privilèges ?

3. Description du contexte

  • Quelle application / URL / paramètre est concerné ?
  • Quel est le rôle de l’utilisateur utilisé ?
  • Quelle est la fonctionnalité visée ?

4. Étapes de reproduction (Step-by-Step)

Cette section est essentielle : elle doit permettre à l’équipe sécurité de reproduire votre faille sans interprétation.

  • 1. Se connecter avec un compte standard
  • 2. Aller sur /profile/settings
  • 3. Modifier le champ bio avec du JavaScript : <script>alert(1)</script>
  • 4. Aller sur le profil public — l’alerte s’exécute

5. POC visuel (captures d’écran / vidéos)

Ajoutez une ou deux captures (ou un lien vers un fichier vidéo), en floutant les infos sensibles.

6. Gravité estimée

Vous pouvez estimer un niveau de gravité selon les critères OWASP ou CVSS, mais restez humble : l'équipe peut ajuster cette valeur.

7. Recommandation

Expliquez brièvement comment le problème peut être résolu. Exemples :

  • Utiliser Content Security Policy (CSP)
  • Échapper les entrées utilisateur côté client et serveur
  • Vérifier l’ownership des ressources côté serveur

Bonnes pratiques pour un bon rapport

  • Soignez votre anglais (c’est souvent la langue utilisée)
  • Numérotez les étapes et illustrez-les si possible
  • Restez factuel, pas de jugement ni de langage agressif
  • Évitez les suppositions techniques : testez et montrez

Exemple de plateforme

  • HackerOne — standard de l’industrie
  • Bugcrowd — programmes publics et privés
  • YesWeHack — en croissance (français + anglais)
  • Intigriti — focus sur qualité des programmes
