Penetration Test Report

Good hacks are invisible. Good reports make them unforgettable.

Introduction

The penetration test report is the final deliverable of an offensive engagement. It’s the only artifact the client will retain after the mission ends. A good report must be readable, well-structured, educational, and actionable by both technical and non-technical audiences.

Key Steps to Writing a Penetration Test Report

1. Cover Page

  • Client name and testing company name
  • Type of test (Blackbox, Greybox, Whitebox)
  • Test date or engagement window
  • Names of the tester(s) or team
  • Confidentiality label: “Strictly Confidential”, “Internal Use Only”, etc.

2. Table of Contents

Use a clickable table of contents with internal links for easier navigation.

3. Executive Summary (Non-Technical)

  • Mission objectives
  • Summary of critical findings
  • Global score (CVSS, risk rating, or matrix)
  • General recommendations

This section is intended for management, avoid deep technical jargon.

4. Methodology

  • Phases: reconnaissance, scanning, exploitation, post-exploitation, reporting
  • Tools used
  • Frameworks and standards (OWASP, PTES, NIST...)

5. Scope and Constraints

  • Authorized IP range
  • Credentials or accounts provided
  • Exact perimeter (infra, application, cloud, etc.)
  • Technical or legal limitations

6. Detailed Technical Findings

Present one vulnerability per section using the following structure:

  • Finding title
  • Severity (Critical / High / Medium / Low / Informational)
  • Clear and concise description
  • Proof of Concept (PoC): command, screenshot, log excerpt
  • Impact: both business and technical
  • Recommendation: practical, technical or organizational fix
  • References (CVE, CWE, OWASP, remediation guides)

7. Conclusion

Summarize the key risks, highlight positive aspects, and encourage implementation of mitigations. You may also include suggestions for follow-up tests or retesting sessions.

Writing Tips

  • Avoid unexplained technical jargon
  • Use proper grammar and aim for clarity
  • Use a clean and professional format (Word, LaTeX, or Markdown to PDF)
  • Adjust vulnerability severity based on the client’s business context
