🛠️ Server-Side Attacks (My Courses)

"If the server trusts input, the attacker shapes output."

🧠 About

Server-side attacks exploit how the backend parses requests, talks to internal services, and builds responses. They often yield high impact : data exfiltration, authentication bypass, and even remote code execution (RCE). This index maps the core families you’ll meet in real engagements and bug bounty.

📘 What is a Server-Side Attack?

  • Source: untrusted input (URL/query/body/headers/files) reaches backend components.
  • Sink: server interpreters, internal networks (metadata, APIs), file systems, or caches.
  • Impact: unauthorized access, data leakage, SSRF pivots, RCE, or full service compromise.

Mindset: trace the data flow : where does input go (parser, protocol, file path, internal HTTP)? What assumptions break?

🎯 Beginner Playbook

  • Probe safely: start with harmless diffs (status/length/time) before heavy payloads.
  • Map sinks: file paths, internal HTTP calls, parsers (XML/template), upload handlers.
  • Chain impact: SSRF → creds/metadata → lateral move; upload → shell → persistence.
  • Log everything: reproduce minimal PoCs, then escalate methodically.

🛡️ Defense Mindset

  • Deny-by-default on parsers (disable XXE, strict content-types, schema validation).
  • Network egress controls + metadata service protections (IMDSv2, no-link-local by default).
  • Safe upload pipelines: type & content validation, store outside webroot, no-exec on upload dirs.
  • Keep dependencies updated; enforce least privilege for services and filesystem access.

"Harden the backend paths : small parsing quirks lead to big breaches."
